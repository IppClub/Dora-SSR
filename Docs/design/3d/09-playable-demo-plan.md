# Dora 3D 完整可玩 Demo 设计与开发跟踪计划

## 1. 文档用途

本文档用于设计、实施和验收 Dora 3D 的首个完整可玩 Demo，同时作为持续更新的开发看板。Demo 的目标不是罗列 API，而是用一段完整游戏流程验证渲染、动画、物理、输入、音频、UI、资源加载和生命周期管理能否共同支撑真实作品。

维护规则：

- 每次开发结束后更新任务状态、验证证据和新发现的问题。
- 状态统一使用 `待开始`、`进行中`、`阻塞`、`完成`。
- 标记完成的任务必须附测试结果、截图、日志或对应提交。
- Demo 暴露出的引擎问题应先建立最小复现，再修复引擎，最后回到 Demo 做回归。
- 范围或技术决策发生变化时，在文末决策记录中保留原因，不直接抹掉历史。

当前状态：`开发中`  
当前里程碑：`macOS 玩家路线验收与修复`  
最后更新：`2026-07-14`

## 2. Demo 定位

### 2.1 游戏概念

暂定名称：**遗迹信使**

形式：10 至 15 分钟的第三人称轻量平台冒险游戏。玩家需要穿越一座小型遗迹，收集三个能量核心，通过物理机关、移动平台和简单敌人区域，最终启动出口。

完整游戏循环：

1. 探索场景并读取空间信息。
2. 移动、跳跃、推动物体或启动机关。
3. 收集能量核心并获得明确反馈。
4. 避开危险或处理简单敌人。
5. 到达出口，展示结算并允许重新开始。

### 2.2 设计原则

- 每项引擎能力必须服务于玩法，不制作独立功能展柜。
- 首轮范围保持为单场景、单角色、单条可完成流程。
- 优先验证引擎组合使用时的稳定性和资源生命周期。
- 美术资源选择统一、轻量、许可清晰的低多边形风格。
- 保留键鼠、手柄和触屏适配空间，不让输入逻辑写死在玩法中。

### 2.3 首版不纳入范围

- 大型开放世界、流式地图和网络同步。
- 复杂战斗 AI、行为树和大量同时活动的角色。
- CSM、点光源阴影、Forward+、GPU morph 和自定义材质编辑器。
- 程序化关卡、存档系统和完整设置菜单。
- 为展示效果而引入无法稳定维护的大型资产包。

## 3. 关卡与流程

| 区域 | 主要玩法 | 验证能力 | 完成条件 |
| --- | --- | --- | --- |
| 起始庭院 | 移动、镜头、跳跃、教学提示 | CharacterController3D、Camera3D、输入、2D UI | 到达遗迹入口 |
| 配重机关 | 推动箱体压住开关 | 动态刚体、静态碰撞体、传感器、接触事件 | 打开第一道门 |
| 断桥区域 | 乘坐移动平台并跨越缺口 | Kinematic body、约束、运动平台角色跟随 | 取得第一个核心 |
| 回廊区域 | 绕过巡逻敌人或触发击退 | 动画、简单状态机、碰撞层、角色交互 | 取得第二个核心 |
| 中央大厅 | 射线操作机关，组合前述机制 | Raycast、picking、点光源、音效 | 取得第三个核心 |
| 出口祭坛 | 插入核心并启动出口 | morph target、动画事件、特效、结算 UI | 完成游戏并可重开 |

失败状态至少包括跌落、受到致命伤害和离开有效区域。失败后应在两秒内回到最近检查点，不重新加载整个引擎实例。

| Sensor / Contact | 开关、拾取物、伤害区 | 进入、停留、离开事件无重复或遗漏 |

| 2D Node / ImGui | HUD、暂停、调试和结算 | 3D 背景与 2D UI 顺序正确 |
| cleanup / cache lifecycle | 重开、检查点和退出 | 无悬空对象、重复释放、持续 RSS 膨胀 |

## 5. 素材方案与治理

### 5.1 候选素材

- [Quaternius Ultimate Platformer Pack](https://quaternius.com/packs/ultimateplatformer.html)：CC0，适合角色、平台和机关原型。
- [Kenney Platformer Kit](https://kenney-assets.itch.io/platformer-kit)：CC0，适合低多边形场景灰盒和最终替换素材。
- [Kenney Interface Sounds](https://kenney.nl/assets/interface-sounds)：CC0，适合 UI 和交互反馈。
- [Poly Haven](https://polyhaven.com/license)：CC0，选择一套尺寸受控的环境图。
- [Khronos glTF Sample Assets](https://github.com/KhronosGroup/glTF-Sample-Assets)：只用于引擎回归或缺陷复现；逐个检查模型许可，不直接作为 Demo 主美术包。

M0 阶段应分别制作角色、环境、动态物体和音频的导入样例，再决定最终素材组合，避免先下载整包后再围绕素材修改玩法。

### 5.2 提交规则

- 不使用 Git LFS，首版新增运行时资源的 Git 体积目标不超过 30 MiB。
- 只提交实际使用且已经优化的文件，不提交完整原始素材包。
- 原始下载文件保存在临时工作目录，转换后的运行时资源进入 Demo 目录。
- 常规纹理优先控制在 512 或 1024 像素；更高尺寸必须记录必要性。
- 模型提交前运行 glTF Validator，并检查坐标、切线、动画、材质和碰撞体生成结果。
- 压缩包必须通过 Dora `Content` 搜索路径和跨平台读取接口处理，不直接依赖系统文件 API。

Demo 目录必须包含 `THIRD_PARTY_ASSETS.md`，每项素材记录名称、作者、来源 URL、许可证、获取日期、采用文件、修改方式和原始包校验值。

## 6. 工程组织

Demo 不放入 `Dora-SSR` 或 `Dora-Example`，而是在 `~/Workspace/Dora` 下新建独立项目仓库。暂定仓库目录为：

```text
~/Workspace/Dora/Dora-Platformer3D-Demo
```

仓库独立管理源码、运行时资源、第三方素材声明、测试和发布记录；Dora-SSR 只保留引擎实现、最小回归案例和本工作文档。Demo 发现的引擎缺陷需要在 Dora-SSR 中补最小复现，不能将只适用于 Demo 的临时修补混入引擎。

建议项目结构：

```text
Dora-Platformer3D-Demo/
  init.ts
  README.md
  THIRD_PARTY_ASSETS.md
  Game/
    Game.ts
    World.ts
    Checkpoint.ts
    Player.ts
    CameraController.ts
    Interaction.ts
    AudioController.ts
  Gameplay/
    Collectible.ts
    PressurePlate.ts
    MovingPlatform.ts
    Hazard.ts
    Enemy.ts
    ExitPortal.ts
  UI/
    HUD.ts
    PauseMenu.ts
    ResultPanel.ts
    DebugPanel.ts
  Assets/
    Model/
    Texture/
    Audio/
    Environment/
    manifest.json
  Test/
    Smoke.ts
    RestartStress.ts
    SceneCycle.ts
```

玩法代码不直接持有底层物理句柄。角色、机关和场景对象负责创建与销毁自身资源，由 `Game` 统一控制关卡状态、暂停、检查点和重开流程。

## 8. 验收标准

### 8.1 功能

- 新玩家无需调试操作即可完成一次流程。
- 键鼠、手柄和触屏至少各有一条完整可用路径。
- 所有检查点、死亡、暂停、重开和退出行为可重复执行。
- 3D 交互不穿透 2D UI，触点坐标和射线方向一致。
- Debug 开关关闭后不改变玩法结果或资源生命周期。

### 8.2 稳定性

- 自动完成流程 20 次无崩溃、卡死或软锁。
- 死亡/重开 100 次后对象注册表回到稳定水平。
- 连续运行 30 分钟，RSS 不呈持续线性增长。
- 场景销毁后 Rust、C++、Jolt 和 bgfx 资源均没有重复释放或残留引用。

### 8.3 性能

- 基准设备为 Apple Silicon macOS Release 构建。
- 正常游戏阶段目标为 60 FPS，P95 CPU 帧时间不超过 16.7 ms。
- 异步加载期间单帧目标不超过 33 ms；超过时必须记录具体资源和阶段。
- GPU finalize、纹理预处理、mesh 上传分别计时，不能只记录总加载时间。
- 资源预算、draw call、活动刚体和活动光源在调试面板中可观察。

### 8.4 视觉与音频

- 固定机位建立起始区、机关区、敌人区和出口四组基线截图。
- PBR 材质在环境图和无环境图模式下均保持可辨识，不出现全黑结果。
- 动画、morph、阴影、透明排序和 normal map 无持续闪烁或明显错位。
- 关键交互均有视觉与音频反馈，空间音频方向与画面一致。

## 9. 问题采集与修复流程

### 9.1 Demo 内诊断面板

调试面板至少显示：

- 当前提交、平台、图形后端和场景状态。
- FPS、CPU/GPU 帧时间、draw call 和可见物数量。
- 活动刚体、接触对、传感器和约束数量。
- ModelCache 命中、异步加载阶段、上传队列和 GPU finalize 时间。
- RSS、场景对象数量和最近一次 cleanup 结果。
- AABB、物理形状、光源、视锥和 picking ray 开关。

### 9.2 问题记录模板


- `P0`：崩溃、数据损坏或无法启动。
- `P1`：无法通关、持续卡死、严重资源泄漏或核心画面错误。
- `P2`：局部功能错误、明显性能问题或可绕过的玩法阻塞。
- `P3`：轻微表现、易用性或文档问题。

### 9.3 修复闭环

1. 在固定检查点稳定复现并收集日志、截图和性能数据。
2. 判断问题属于 Demo 逻辑、素材数据还是引擎实现。
3. 引擎问题建立最小测试，不直接只在完整 Demo 中试错。
4. 修复后依次运行最小测试、受影响区域和完整流程。
5. 更新本计划的任务状态、问题记录和基线证据。
