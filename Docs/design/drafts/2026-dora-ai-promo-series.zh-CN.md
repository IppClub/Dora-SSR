## 1. 这一年，Dora 引擎开始由 AI 参与开发

# 这一年，Dora 引擎开始由 AI 参与开发

## 不是把 AI 当成一个宣传功能

&emsp;&emsp;如果要给今年 Dora SSR 的开发找一个最大的主题，我现在会毫不犹豫地说是 AI。不是因为 AI 很热，所以一定要在宣传里蹭一下；而是因为今年 Dora 引擎相关的开发，实际上已经很大程度上由 Codex 参与完成。

&emsp;&emsp;这件事比“Dora 内置了一个 Coding Agent”更重要。因为它不是把 AI 当成用户界面上的一个新按钮，而是把 AI 放进了 Dora 自己的真实开发流程里。Dora 的 Agent、Web IDE、CLI、TypeScript 工具链、DoraX、Android 和 Linux 适配、发布链路，很多工作都在这种人类维护者和 AI 编程助手协作的状态下推进。

&emsp;&emsp;以前我们讲 Dora SSR，常常会讲跨平台、Web IDE、多语言、真机开发。这些都仍然成立。但今年以后，这些能力应该放在一个新的背景里理解：如果 AI 开始成为项目开发的主要劳动力之一，游戏引擎的工具链应该长成什么样？

## AI 真的进入工程以后，问题才开始

&emsp;&emsp;AI 写代码的演示很容易做。给一个需求，生成一段函数，跑通一个 demo，视频里效果很好。但真实工程不是这样。真实工程里有生成代码，有历史兼容，有构建脚本，有平台差异，有 CI，有发布包，有用户已经改了一半但不能乱碰的文件。

&emsp;&emsp;今年使用 Codex 开发 Dora 时，最有价值的地方并不是它一次能写多少代码，而是它能持续进入这些工程细节里：先读当前 checkout，不相信过期记忆；改 TS 源码后跑 Dora 的 TS 构建，让 Lua 生成物保持同步；处理 Android WebView 里的粘贴问题；追 Linux ARM64 设备上的输入焦点；修发布流程里版本元数据和镜像同步的细节。

&emsp;&emsp;这才是 AI 编程开始变得严肃的地方。它不再是一个会写代码片段的玩具，而是在真实项目里承担重复、琐碎、但必须做对的工程劳动。

## Dora 自己也在被 AI 改造

&emsp;&emsp;有意思的是，Dora 不只是被 AI 开发，它自己也在变成一个更适合 AI 参与的开发环境。我们给 Dora 加 Coding Agent，不只是为了让用户在 Web IDE 里问问题，而是希望 Agent 能理解项目、使用工具、读文件、跑构建、修错误、保留上下文。

&emsp;&emsp;这两件事互相影响。Codex 参与开发 Dora，让我们更清楚 AI 在工程里会在哪里犯错；这些经验又反过来推动 Dora 内置 Agent 的设计。比如工具调用要能并行但不能乱序，生成文件不能手写乱改，验证命令要尽量收窄到相关目录，失败日志要能被带回 Agent 形成下一步行动。

&emsp;&emsp;所以今年 Dora 的 AI 主题，不是外部贴上去的一层功能，而是从项目开发方式一路长到用户开发体验里的。

## 小项目为什么更需要 AI

&emsp;&emsp;大型商业引擎有专门的渲染工程师、工具工程师、构建工程师、文档团队和测试团队。Dora 这样的开源小项目没有这些奢侈配置。过去很多事情不是不知道该做，而是没有足够的人手去做。

&emsp;&emsp;AI 改变的正是这个成本结构。它不能替代项目方向，也不能替代维护者的判断，但它可以把大量“知道该做但没人有空做”的工作推进起来。修文档、查构建错误、补工具链、整理发布流程、追跨平台小问题，这些事情被 AI 承担一部分以后，小项目的上限会被抬高。

&emsp;&emsp;这也是 Dora SSR 今年最值得讲的故事：一个资源有限的开源游戏引擎，开始认真把 AI 当成工程协作者，而不是宣传噱头。

## 2. 游戏引擎里的 Coding Agent

# 游戏引擎里的 Coding Agent：AI 不只是聊天窗口

## AI 辅助开发不能只停在“你问我答”

&emsp;&emsp;现在很多开发工具都会接 AI。最常见的形态是在 IDE 里放一个聊天窗口，开发者问问题，AI 回答。这个形态有用，但它离真实工程协作还有距离。

&emsp;&emsp;真实工程开发不是只回答问题。它要知道项目目录在哪里，哪些文件是源文件，哪些文件是生成物，改完应该跑什么命令，失败以后该读哪段日志，用户当前工作区里哪些改动不能碰。AI 如果不知道这些，就很容易变成一个看似聪明但经常添乱的助手。

&emsp;&emsp;Dora 的 Coding Agent 想解决的就是这个问题。它不是 Dora Web IDE 旁边的装饰聊天框，而是游戏开发工具链的一部分。

## Agent 要能使用工具，也要懂规矩

&emsp;&emsp;今年 Dora 的 Agent 相关开发集中在工具能力和工程约束上。它需要能读写文件、搜索项目、调用构建、处理日志、维护会话、使用技能说明，也需要支持流式工具执行、批量工具调用、记忆压缩和失败恢复。

&emsp;&emsp;但工具越多，越需要规矩。Agent 不能看见错误就全项目乱改，不能把生成 Lua 当作唯一源头手写，不能因为一个诊断噪声就改掉无关模块。它要学会在 Dora 的项目结构里行动。

&emsp;&emsp;这和训练一个新同事很像。你不是只给他一把锤子，而是要告诉他哪些墙能敲，哪些管线不能碰，敲完以后要检查什么。

## 游戏开发里的 AI 应该贴近项目

&emsp;&emsp;游戏开发有大量任务不需要惊天动地的创造力，但很消耗人：查 API、补样板代码、修资源路径、解释构建失败、生成简单组件、整理教程、检查平台打包。Dora 的 Agent 如果能把这些事情做掉一部分，独立开发者和小团队就能把更多精力留给游戏本身。

&emsp;&emsp;我不认为 AI 应该替代创作者设计游戏，也不认为它能凭空理解一个游戏应该好不好玩。AI 更适合先成为项目里的工程助手。它把创作者从重复劳动里解放一点点，这件事已经很有价值。

## Dora 的 Agent 是给未来工具链探路

&emsp;&emsp;Dora SSR 本来就不是一个只有运行时的引擎。它有 Web IDE，有语言工具链，有内容编辑器，有 CLI，有跨平台发布路径。把 Agent 放进这个环境里，才会产生真正有意思的东西。

&emsp;&emsp;未来的游戏引擎可能不只是提供 API 和编辑器，还会提供一个能理解项目、能调用工具、能跟随开发者上下文工作的智能层。Dora 的 Coding Agent 还在早期，但方向已经很清楚：AI 要进入工具链，而不只是停在聊天框。

## 3. 为什么我们把焦点收向 AI + TypeScript

# 为什么我们把焦点收向 AI + TypeScript

## 多语言时代的新问题

&emsp;&emsp;Dora SSR 一直支持很多语言。Lua、TypeScript、TSX、Teal、YueScript、Wa、Rust、C#，这些语言给了开发者很多入口。过去我会把这件事看成 Dora 的开放性：不同开发者可以按自己的习惯进入引擎。

&emsp;&emsp;但 AI 参与开发以后，问题开始变化。对人来说，语言审美和习惯很重要；对 AI 来说，训练数据规模、类型信息、工具链反馈、错误诊断质量、生态文档密度同样重要。AI 能不能稳定地写出可维护代码，很大程度上取决于语言和工具链能不能给它足够明确的约束。

&emsp;&emsp;从这个角度看，TypeScript 变得越来越重要。

## TypeScript 是目前最适合 AI 协作的解

&emsp;&emsp;TypeScript 的优势不只是“有类型”。它有成熟的语言服务，有大量真实项目语料，有清晰的错误诊断，有 Monaco 和编辑器生态，有 LLM 很熟悉的语法模式。对 AI 来说，这些都是非常重要的轨道。

&emsp;&emsp;Dora 通过 TypeScriptToLua 把 TS/TSX 转成 Lua，既能利用 Dora 现有的 Lua 运行基础，又能给开发者和 AI 一个更结构化的编程表面。AI 写 Lua 也可以，但缺少类型约束时，它更容易在 API 名称、返回值形状、对象生命周期上犯错。TypeScript 至少能把一部分错误提前暴露出来。

&emsp;&emsp;所以我们现在越来越倾向于把 AI + TypeScript 看成 Dora 当前的最优解。不是因为其它语言不重要，而是因为在“让 AI 稳定参与游戏工程开发”这个目标下，TypeScript 的综合收益最高。

## TSX 和 DoraX 的意义也变了

&emsp;&emsp;以前看 DoraX，重点可能是“能不能用 TSX 写 2D 场景”。这是一个很有趣的前端范式迁移。但在 AI 时代，它还有另一个意义：TSX 给场景结构提供了更明确的文本表达。

&emsp;&emsp;AI 很擅长生成和修改结构化文本。一个菜单、一个配置面板、一个简单场景，如果用 TSX 描述，AI 更容易读懂层级关系，也更容易局部修改。相比散落在命令式代码里的节点创建和属性赋值，声明式结构更适合 AI 做维护。

&emsp;&emsp;这并不意味着所有游戏逻辑都要 TSX 化。游戏运行时仍然有大量状态、物理、动画和性能敏感路径。但对于 UI、工具界面、场景搭建、教学示例，TSX 可能会成为 Dora 面向 AI 协作的一个重要表达层。

## 聚焦不是放弃多语言

&emsp;&emsp;把焦点收向 AI + TypeScript，不等于 Dora 要放弃其它语言。Lua 仍然是运行时基础，YueScript 仍然有它的表达力，Wa/Rust/C# 仍然适合扩展和不同生态用户。

&emsp;&emsp;但项目需要主路径。过去主路径可能是“人类开发者选择自己喜欢的语言”；未来主路径可能会变成“AI 和人类都能稳定理解、修改、验证的语言”。在当前条件下，这条路径更像 TypeScript。

&emsp;&emsp;这不是语言信仰问题，而是工程效率问题。

## 4. 多语言体验会不会被 AI 抹平

# 多语言体验会不会被 AI 抹平

## 以前多语言解决的是人的入口问题

&emsp;&emsp;Dora 支持多语言，最初解决的是人的入口问题。有人喜欢 Lua 的轻量，有人喜欢 TypeScript 的类型，有人喜欢 YueScript 的简洁，有人想用 Wa 或 Rust 做扩展。不同语言代表不同审美、不同背景和不同开发习惯。

&emsp;&emsp;在没有 AI 的时候，这种差异很重要。因为写代码的人就是开发者本人。语言越顺手，创作越自然。Dora 给不同语言留入口，就是给不同人留入口。

&emsp;&emsp;但 AI 进入以后，一个新的问题出现了：如果大量代码由 AI 生成和维护，语言体验还会像过去一样重要吗？

## AI 可能会抹平一部分语言门槛

&emsp;&emsp;对很多用户来说，未来他们可能不再需要完全掌握某门语言，才能用它写游戏。用户描述需求，AI 生成代码；用户看不懂细节，AI 解释；代码报错，AI 修复。这样一来，不同语言之间的门槛会被抹平一部分。

&emsp;&emsp;比如一个用户不熟悉 Lua，但如果 Agent 能稳定生成 Dora Lua 代码，并解释每一段逻辑，那么 Lua 的学习成本就下降了。一个用户不熟悉 TypeScript，但如果 AI 能根据 Dora API 和类型提示持续修正，TS 的门槛也会下降。

&emsp;&emsp;从这个角度看，多语言支持的价值会发生变化。它不再只是“用户亲自写哪种语言”，也可能变成“AI 能把用户意图翻译到哪种语言表面上”。

## 但 AI 也会放大主路径优势

&emsp;&emsp;不过，AI 抹平语言门槛，并不代表所有语言在工程里会变得完全等价。恰恰相反，AI 会放大主路径语言的优势。

&emsp;&emsp;训练数据更多、类型信息更完整、错误诊断更清晰、工具链更成熟的语言，会让 AI 更稳定。AI 能写很多语言，但稳定维护一个长期项目时，它仍然需要反馈回路。TypeScript 在这方面显然有优势。

&emsp;&emsp;所以 Dora 的多语言未来可能会变成这样的结构：多语言继续作为生态入口存在，但 AI 协作的推荐路径会越来越集中到 TypeScript/TSX。其它语言不是被淘汰，而是变成特定场景、特定用户和特定扩展层的选择。

## 多语言从“开发者选择”变成“AI 翻译层”

&emsp;&emsp;我觉得更有想象力的方向是：Dora 的多语言能力以后不只是给人手写，也给 AI 做翻译层。用户可以用自然语言描述玩法，Agent 根据项目情况选择生成 TypeScript、Lua 或其它语言；也可以把旧 Lua 逻辑解释成 TS 版本，把 TSX 场景拆成 Lua 运行逻辑，把底层扩展引导到 Wa 或 Rust。

&emsp;&emsp;这样看，多语言不是 Dora 的历史包袱，而是 AI 时代的素材库。只是我们需要承认，在所有这些语言表面中，目前 TypeScript 最适合作为 AI 协作的主干。

&emsp;&emsp;多语言不会消失，但它的意义会改变。

## 5. Web IDE 在 AI 时代的新位置

# Web IDE 在 AI 时代的新位置

## Web IDE 不只是浏览器里的编辑器

&emsp;&emsp;以前讲 Dora 的 Web IDE，重点通常是跨设备：引擎在目标设备上跑，开发者用浏览器接入。这个设计仍然重要，因为它让 Android、Linux、掌机等设备都可以成为开发现场。

&emsp;&emsp;但 AI 时代以后，Web IDE 的意义又多了一层。它不只是代码编辑器，而是 AI 能看到项目、操作项目、验证项目的工作台。

&emsp;&emsp;如果 AI 要真正参与开发，它需要的不只是一个输入框。它需要文件树、编辑器、日志、构建结果、资源预览、项目状态、历史上下文。Web IDE 正好是这些信息汇聚的地方。

## AI 需要上下文，Web IDE 提供上下文

&emsp;&emsp;AI 最怕没有上下文。没有上下文，它就只能根据常识猜；猜得越多，越容易错。游戏项目尤其如此，因为很多错误不在单个函数里，而在资源路径、运行平台、构建方式、生成文件和项目约定之间。

&emsp;&emsp;Dora Web IDE 里接入 Agent，就可以把这些上下文逐步喂给 AI。比如当前打开的文件、当前项目根、构建日志、TypeScript 诊断、Dora API 声明、文件预览、Git 状态。这些信息合在一起，才让 AI 有机会做出正确操作。

&emsp;&emsp;所以 Web IDE 的价值从“给人写代码”扩展成了“给人和 AI 一起工作”。

## 内容编辑器也可以成为 AI 的操作对象

&emsp;&emsp;今年 Dora Web IDE 接入了 Action Editor、Body Editor、Particle Editor、Yarn Editor 等工具。过去我们会把它们理解成面向人的内容编辑器。未来它们也可能成为 AI 的操作对象。

&emsp;&emsp;AI 可以根据描述生成对白草稿，帮你调整配置，解释动画资源结构，检查碰撞形状是否合理，或者把教程里的示例改成当前项目的资源路径。前提是这些工具和数据都在 Dora 的项目环境里，而不是散落在外部软件中。

&emsp;&emsp;这就是 Web IDE 的长期价值：它把游戏创作的不同材料聚到一个 AI 可以理解和操作的空间里。

## 移动端和真机开发仍然重要

&emsp;&emsp;AI 不会让真机开发变得不重要。相反，AI 生成的东西更需要真实运行反馈。代码能不能在 Android WebView 里正常输入，手柄焦点有没有穿透，Linux ARM64 设备能不能启动，发布包能不能安装，这些都不是模型凭空想象能保证的。

&emsp;&emsp;Dora 的 Web IDE + 真机运行结构，刚好能给 AI 一个更短的反馈回路。生成代码以后，在目标设备上跑；报错以后，日志回到 Agent；Agent 再修。这比“AI 写完代码，人类手动复制到另一个环境里试”要自然得多。

&emsp;&emsp;所以 AI 时代的 Dora Web IDE，不只是编辑器，而是反馈回路。

## 6. 小型开源引擎的 AI 生存方式

# 小型开源引擎的 AI 生存方式

## 小项目最缺的不是想法

&emsp;&emsp;小型开源项目通常不缺想法。真正缺的是持续把想法做完的人手。文档要写，CI 要修，平台要适配，用户反馈要处理，发布包要同步，旧接口要兼容，新功能还要继续推进。

&emsp;&emsp;Dora SSR 今年的经验让我感觉到，AI 对小项目最直接的价值，不是让它突然拥有天才设计，而是让它有机会完成更多本来应该做、但过去没精力做的事情。

&emsp;&emsp;这是一种很朴素的生产力变化。

## Codex 像一个不知疲倦但需要规训的工程助手

&emsp;&emsp;用 Codex 开发 Dora 的过程里，我最大的感受不是“AI 无所不能”，而是“AI 很能干，但必须被规训”。它可以读代码、查文件、写补丁、跑命令、追日志，连续处理很多细碎问题。但它也必须遵守项目规则，不能乱改无关文件，不能相信旧假设，不能跳过验证。

&emsp;&emsp;这就要求项目本身也变得更清楚：验证命令要明确，生成路径要明确，文档要说明哪些文件是源头，技能和工作流要沉淀下来。AI 不是只改变写代码的人，它也会反过来要求项目变得更适合被理解。

&emsp;&emsp;这对 Dora SSR 是好事。一个能被 AI 稳定操作的项目，通常也更容易被新人理解。

## AI 会改变开源项目的维护边界

&emsp;&emsp;过去很多开源项目的维护边界由人力决定。维护者没有时间，就不做；平台太麻烦，就先放着；文档太多，就以后再补。AI 参与以后，这条边界会往外推一点。

&emsp;&emsp;Dora 今年能同时推进 Agent、Web IDE、CLI、DoraX、Android、Linux、发布流程，很大程度上就是因为 AI 分担了大量具体工程执行。维护者仍然要判断方向、审查结果、决定取舍，但执行成本明显变低。

&emsp;&emsp;这不意味着以后开源项目都可以不用人。恰恰相反，人的判断会更重要。因为 AI 越能执行，越需要有人决定什么值得执行。

## Dora 的答案是把 AI 放进工作流

&emsp;&emsp;我不想把 Dora SSR 包装成一个“AI 游戏引擎”。这个说法太空。Dora 更实际的路线是：把 AI 放进开发工作流里。先让 AI 参与引擎自身开发，再把这些经验转化为 Web IDE 里的 Coding Agent、TypeScript 主路径、CLI 验证和项目技能。

&emsp;&emsp;这条路线不炫，但可持续。它不是说 AI 会替你做出好游戏，而是说 AI 可以帮你更快越过工具链、样板代码、构建错误和平台细节，把精力留给游戏创作。

&emsp;&emsp;对 Dora SSR 这样的开源游戏引擎来说，这可能就是最现实、也最有价值的 AI 方向。
