# 游戏引擎里的 Coding Agent：懂现场，比会动手更难

> 「Dora SSR 2026 开发纪事」系列 · 第 2 篇 / 共 10 篇

## 当“能做事的 AI”已经不稀奇

&emsp;&emsp;放到两三年前，“一个能动手做事的 AI”还是个挺新鲜的说法。那时候大家在开发工具里碰到的 AI，多半是一个聊天窗口——问 API，它解释；贴报错，它猜原因；让它写函数，它吐一段代码。有用，但停留在“问答”那一层。一个只回答问题的 AI，说得头头是道，一动手就踩到项目的暗线：哪些文件是源头、哪些是生成物，哪条命令能验证最小范围，哪个失败其实只是本地服务没起来。

&emsp;&emsp;但现在是 2026 年。能读仓库、改代码、跑测试、自己定位失败的 coding agent 早已不稀奇——Cursor、Claude Code、Codex 自己，每天都在无数项目里干活。“AI 能做事”已经从卖点变成了基础设施。所以今天再谈引入 agent，问题早就不是“它能不能动手”，而是“它懂不懂我这个项目的现场”。通用 agent 知道 git 和 TypeScript 的通用规矩；却不知道 Dora 里一段 TypeScript 最后会被转译成 Lua，不知道一个 Node 该怎么继承扩展，不知道碰撞体多边形为什么会分解失败，更不知道这时候该跑哪条最小验证命令。

&emsp;&emsp;Dora SSR 做 Coding Agent，出发点就在这里：不是再做一个通用 agent——那种工具开发者手里早就有一把——而是让 agent 住进 Dora 的环境里，先懂这个引擎的脾气，再谈帮创作者干活。读项目文件、调用工具只是基本盘；它还得理解 Dora 的构建方式和资源格式，记住项目规则，在失败以后继续追踪，而不是把“建议你检查一下路径”当作终点。

## 控制室里的副驾驶

&emsp;&emsp;我更愿意把 Dora 的 Coding Agent 想象成控制室里的副驾驶。副驾驶不是船长，不能替创作者决定游戏应该长什么样；但它应该看得懂仪表，知道哪盏灯亮了意味着什么，知道什么时候可以同时检查几条线路，什么时候必须等上一条命令结束。它不是为了把人挤出驾驶位，而是为了让创作者少被机械性的工程事务拖住。

&emsp;&emsp;这个想法听起来温和，做起来却非常具体。Agent 需要知道工具柜里有哪些工具，每个工具的参数是什么，哪些工具只读，哪些工具会改文件，哪些可以并行运行，哪些需要谨慎等待。它还要知道自己当前扮演什么角色，是主 Agent，还是被派出去调查某个局部问题的子 Agent；它要能把长会话压缩成记忆，又不能在压缩以后忘掉关键约束。

&emsp;&emsp;这也是为什么 Dora 的 Agent 系统后来逐渐长出了工具注册表、会话状态、批量工具调用、XML 决策修复、记忆摘要、任务状态卡片等结构。它们看起来像一串枯燥的工程名词，其实都是为了回答同一个问题：怎样让 AI 不只是聪明，而是可靠？

## 可靠比聪明更难

&emsp;&emsp;做 Coding Agent 最容易被低估的，是“可靠”这件事。模型能写出一段漂亮代码并不难，难的是它能不能在一个真实项目里持续做正确的小事。它能不能先读源码再判断？能不能在用户说“不要碰未暂存文件”时真的守住边界？能不能知道 Dora 的 TypeScript 最终会转成 Lua，因此验证路径不能照搬普通 Web 项目的习惯？能不能在看到生成文件不一致时回到上游声明，而不是直接改生成物？

&emsp;&emsp;Dora 项目里的很多 Agent 设计，都是被这些真实问题逼出来的。比如工具调用要分清楚“可以并行的查阅”和“必须串行的修改”；模型输出的 XML 决策可能不完整，修复器要帮它恢复格式，却不能替它重新做决定；会话太长时需要压缩，但压缩不能把“哪些目录是项目记忆、哪些文件是用户源代码”这种边界弄丢。

&emsp;&emsp;这有点像训练一个新同事。你不只是告诉他“这里有锤子和扳手”，还要告诉他哪些墙不能拆，哪些线不能剪，哪张图纸已经过期，哪条检查命令最能说明问题。AI 的不同之处在于，它可以很快阅读大量内容，也可以很快犯下大量看起来合理的错误。所以工程系统必须给它铺轨道。

## 游戏引擎为什么需要自己的 Agent

&emsp;&emsp;也许有人会问：既然已经有通用 AI 编程工具，为什么游戏引擎还要内置自己的 Coding Agent？我的答案是，游戏引擎不是普通代码库，它同时包含运行时、编辑器、资源格式、脚本语言、绑定生成、平台适配和项目习惯。通用工具可以帮忙，但它不会天然知道 Dora 的纹理、节点、动作、TSX、YueScript、Teal、Web IDE、CLI 和设备工作流之间如何连接。

&emsp;&emsp;Dora SSR 本身就不是一个单薄的运行库。它有浏览器里的 IDE，有多语言构建入口，有资源编辑器，有本地 Web 服务，有 CLI，有跨平台运行时。开发者已经在 Dora 的环境里写代码、看日志、跑构建，那么让 Agent 也在这个环境里工作，就不是强行叠加功能，而是把原本分散的工具经验汇合起来。

&emsp;&emsp;游戏开发里有太多夹在创作和工程之间的任务。你想做的是一个玩法原型，但突然要查某个 API；你想调一个动作曲线，但资源格式报错；你想把一段 Lua 示例改成 TypeScript，却被类型声明卡住；你想在掌机上试一个输入逻辑，却发现焦点被工具窗口抢走。这些任务都不应该由创作者独自反复啃。Agent 的价值就在于，它能陪你把这些中间地带走过去。

## AI 不是魔法，而是工作流

&emsp;&emsp;今年 AI 编程最容易被包装成魔法：一句话生成项目，一次提示完成应用，几分钟替代团队。Dora 的经验反而更朴素。真正让 AI 变得有用的，不是把它想象成无所不能，而是把它放进工作流。给它项目规则，给它工具，给它验证方式，给它记忆，也给它限制。

&emsp;&emsp;在 Dora 这样的开源项目里，这种工作流价值尤其明显。小项目最缺的往往不是创意，而是持续处理琐碎工程债的时间。文档要同步，声明要检查，平台差异要确认，发布资产要回读，Web IDE 的小体验要修，构建失败要沿着日志追。AI 如果能稳定参与这些事情，就不只是提高一点编码速度，而是在改变项目能承受的维护密度。

&emsp;&emsp;这也是 Dora Agent 的长期方向：它不应该被设计成一个“写代码按钮”，而应该成为理解 Dora 的工程伙伴。它要知道这个引擎的脾气，知道创作者经常被什么问题打断，知道哪些检查能给出真实答案。只有这样，AI 才会从聊天窗口变成开发环境的一部分。

## 一次记忆位置的小事故

&emsp;&emsp;Agent 的会话记忆也有一个很小但很真实的故事。系统里需要保存 `SESSION_SUMMARY.md` 这类长会话摘要，让 Agent 在上下文被压缩以后仍然能接着工作。最开始问题并不显眼：主会话的记忆文件有时落在 `.agent/`，有时落在 `.agent/main/`，路径看起来只是差了一层目录，但对长期运行的 Agent 来说，这就是“日记本放在两个抽屉里”。

&emsp;&emsp;如果只是补一个迁移脚本，把旧位置文件搬到新位置，表面上也能解决。但用户明确指出，迁移和清理旧文件的逻辑可以去掉，修复应该收窄到存储边界。这句话很重要。它把问题从“怎样把眼前文件整理干净”拉回到“为什么文件会放错地方”。最后的方向是在存储层加一道归一化：只要传入的 scope 是空字符串，就统一落到 `main`，让主会话记忆永远固定在 `.agent/main/SESSION_SUMMARY.md` 这一个命名空间里。

&emsp;&emsp;这个小事故很像 Agent 设计的缩影。AI 系统最怕把补丁叠成补丁。真正稳的做法，是找到边界，把规则放在边界上。记忆文件的位置不是用户关心的功能，却决定了长会话能不能可靠延续。一个会工作的 Agent，首先要有一本不会丢页、不会放错抽屉的工作日志。

&emsp;&emsp;也正因为这些机制——工具注册表、批量调用、XML 修复、记忆压缩——大多不是凭空设计，而是在 Codex 真实处理 Dora 任务时被一个个具体失败逼出来的，它才不像一个装饰性的 AI 面板，而更像从真实工程劳动里长出来的工具。每个看似抽象的约束背后，都有一次"工具拿错了""上下文压缩后忘掉边界了""生成文件和源文件对不上了"的教训。可靠，就是这样一点点磨出来的。
