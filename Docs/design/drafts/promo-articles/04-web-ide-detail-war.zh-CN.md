# Web IDE 的细节战：工具好不好用，藏在红线和焦点里

> 「Dora SSR 2026 开发纪事」系列 · 第 4 篇 / 共 10 篇

## 细节不是装饰，是工具的摩擦系数

&emsp;&emsp;开发工具的好坏，很多时候不是由最大功能决定的，而是由最小摩擦决定的。一个按钮偏一点，一个错误提示含糊一点，一个后台诊断太吵一点，一个输入焦点跑错一次，都会让创作者从正在想的游戏里掉出来。Dora Web IDE 今年修的很多问题，表面看都是细节；但对真正每天使用工具的人来说，这些细节就是工具的摩擦系数。

&emsp;&emsp;摩擦系数高的工具不会立刻不能用，它只是让每一步都多一点阻力。你写代码时多等一秒，切换文件时多看几条无关红线，粘贴密钥时失败一次，调试资源时不知道哪行错了。这些阻力单独看不严重，叠在一起就会消耗创作热情。

&emsp;&emsp;Dora SSR 的 Web IDE 不是大型商业 IDE，它没有无限资源去堆功能。因此更需要把有限精力放在“最常被碰到的摩擦”上。让错误更准，让诊断更安静，让输入更可靠，让设备上的界面更顺手，这些就是小工具变得可信的方式。

## 红线应该像铅笔，不应该像警报器

&emsp;&emsp;TypeScript 实时诊断的调整，是一个很好的例子。最开始的直觉很简单：编辑器当然应该帮用户检查 TypeScript，越多越安全。但 Web IDE 里打开多个模型、切换多个项目、加载内置库和项目库以后，Monaco 的自动诊断可能开始检查并非当前关注的内容。于是你明明在改一个文件，却被其它模型的红线影响判断。

&emsp;&emsp;用户后来把方向说得很清楚：编辑时检查只走 Monaco 的异步 worker，并且只关注当前活动 TS/TSX 文件；真正需要 TSTL 的检查，放到保存、转译或构建路径里。这个决定背后有一个很重要的小观点：实时反馈应该像铅笔标注，轻、快、贴着当前视线；构建反馈才像正式质检，严格、完整、不怕慢。

&emsp;&emsp;如果把两者混在一起，工具就会像过度敏感的警报器。你只是挪动一把椅子，它把整栋楼的消防系统都拉响。Dora Web IDE 的调整不是减少正确性，而是让正确性分层。写代码时让人保持流动，构建时再给出更完整的判断。

## 从“检查所有”到“只照当前”

&emsp;&emsp;这个调整过程也很像一次小型侦查。先发现编辑器和实际转译器的路径规则不一致，Monaco 看到的项目根和 TranspileTS 使用的项目根不是同一套语义；于是补齐项目 root 和内置库的查找规则。接着又发现，路径一致以后，诊断范围仍然可能太大。于是把全局自动诊断关掉，再自己触发当前活动模型的检查。

&emsp;&emsp;中间还出现过一次典型偏差：关掉全局诊断以后，实时检查也没了。用户指出这个问题，方向才进一步明确：不是不要实时诊断，而是不要后台模型诊断。于是实现变成了“只照当前文件的一盏台灯”。当前模型改变时检查，内容变化时防抖检查，异步结果回来时还要确认它仍然属于当前文件，避免旧结果覆盖新状态。

&emsp;&emsp;这个故事非常适合说明开发工具设计里的边界感。一个功能开关很容易从“太吵”变成“太安静”，真正好的状态在中间：不检查无关内容，但对当前创作对象保持敏感。

## Android WebView 里的粘贴和按钮

&emsp;&emsp;Web IDE 的另一个细节战发生在 Android 设备上。浏览器里的编辑器到了 WebView 里，很多桌面习惯都会失效或变形。粘贴 LLM 配置、Git 密钥、Agent prompt 这类文本，如果在 Android WebView 里不可靠，AI 功能和协作功能就会被卡在第一步。浮动按钮如果位置不对，会挡住滚动条或输入区域。软键盘弹出时，布局也可能突然变得别扭。

&emsp;&emsp;这类问题最容易被忽略，因为它们不是“引擎核心”。但从用户角度看，输入密钥失败就是工具失败，按钮挡住内容就是工具失败。开发工具不是只在编译时才存在，它存在于每一次点击、粘贴、滚动和聚焦里。

&emsp;&emsp;Dora 把 Web IDE 放进浏览器，也放进设备上的 WebView，就必须接受这些设备细节。这里没有一劳永逸的抽象，只有一次次把具体摩擦磨低。

## 资源错误也要说人话

&emsp;&emsp;除了代码红线，资源错误也需要“说人话”。粒子 XML 出错、碰撞体形状不合法、Yarn 数据没准备好，如果工具只说失败，开发者就得在文件格式里自己找针。Dora 的资源编辑器逐渐补充更具体的诊断，就是为了把错误从“黑箱失败”变成“可修问题”。

&emsp;&emsp;这对 AI 也很重要。未来 Agent 参与项目时，它同样依赖工具反馈。如果诊断只是一句 Unknown error，AI 只能猜；如果诊断说某个属性不是数字、某个多边形自相交、某个文件缺少必要数据，AI 就可以沿着明确线索修。工具为人说清楚，也是在为 AI 铺路。

&emsp;&emsp;所以 Web IDE 的细节战不是界面洁癖，而是 Dora 把开发环境变成可协作现场的基础。人类和 AI 都需要清晰、及时、不过度的反馈。

## 一个未知文本文件的边界

&emsp;&emsp;Web IDE 还有一类不起眼的体验：用户打开一个工具不认识的文本文件时应该怎么办。最粗暴的做法是拒绝打开，最冒险的做法是把任何东西都当作可编辑文本。Dora 后来处理过未知文本文件回退到 `.txt` 编辑的逻辑，同时又要防止文件夹被误当文件打开。这听起来很小，但它关乎用户对工具的信任。

&emsp;&emsp;一个开放项目里，用户总会放进一些工具原本没预料到的文件。配置片段、笔记、临时脚本、外部资源说明，都可能出现在项目目录里。Web IDE 如果太狭窄，用户会被迫离开；如果太随意，又可能把不该编辑的对象当文件处理。最后合理的边界是：后端确认它确实是文件，前端再给一个普通文本入口。

&emsp;&emsp;这个故事和 TS 红线、WebView 粘贴、资源诊断其实是同一类问题。工具要开放，但不能糊涂；要方便，但不能越界。很多好用的感觉，就来自这些看不见的边界判断。

## 细节战最后拼成的是信任

&emsp;&emsp;当我们把这些小事摆在一起，会发现 Web IDE 的细节战并不是零散修补。实时诊断不打扰当前文件，资源错误讲清楚，Android 输入能粘贴，未知文本能安全打开，按钮不挡滚动条，Agent 能读到明确错误。这些事情共同塑造了一种信任：开发者相信这个工具不会随便打断他，也相信它出声时是有理由的。

&emsp;&emsp;信任对创作工具尤其重要。游戏开发本来就充满不确定，玩法、节奏、视觉、输入手感都要反复试。如果工具本身也不稳定，开发者就会把精力花在防备工具上。Dora Web IDE 的目标不是让每个界面都显得庞大，而是让开发者逐渐感觉：这里可以放心写，放心试，放心让 AI 一起查。
