# DoraX 和 TSX：把场景树写成一张透明的描图纸

> 「Dora SSR 2026 开发纪事」系列 · 第 5 篇 / 共 10 篇

## 为什么游戏场景也需要一种“可读的形状”

&emsp;&emsp;游戏引擎里的场景树，常常是一种看得见却不容易读的东西。它在运行时很清楚：节点挂在哪里，子节点谁先谁后，动作在哪个对象上播放，某个 UI 元素何时被销毁。但在代码里，这些关系经常被一串命令式调用拆散。先创建节点，再设置属性，再 addChild，再绑定事件，再启动动作。逻辑没错，只是读起来像看一堆零件摆在桌上，需要在脑子里重新拼出形状。

&emsp;&emsp;DoraX 引入 TSX 的意义，不是为了追逐前端潮流，而是为了让场景结构重新变成可读的形状。TSX 像一张透明的描图纸，铺在 Dora 的场景树上。你写下来的代码，不只是“执行了哪些步骤”，也尽量接近“这个场景长什么样”。这对游戏开发很重要，因为游戏对象之间的层级关系、状态关系和生命周期关系，往往就是玩法的一部分。

&emsp;&emsp;当然，把 TSX 带进游戏引擎并不等于把 React 搬进来。DoraX 面对的是 Dora 自己的节点、动作、调度、资源和运行时约束。它要借用声明式表达的好处，又不能假装游戏场景和网页 DOM 完全一样。这中间有很多细小的取舍，而今年的 DoraX 开发，正是在这些取舍里慢慢变得稳固。

## 声明式不是少写代码，而是少在脑子里拼装

&emsp;&emsp;很多人理解声明式 UI 时，会把重点放在“代码更短”。但 DoraX 更重要的价值不是少几行，而是少一些脑内拼装。你看到一个 TSX 结构，就能大致知道场景里有哪些节点，父子关系是什么，哪些属性是静态的，哪些来自状态，哪些子节点会随着条件变化。这比在一堆 addChild 和 removeFromParent 之间追踪对象生命要直观得多。

&emsp;&emsp;在游戏开发里，这种直观会影响创作速度。一个角色血条、一个暂停菜单、一个对话框、一个调试面板，如果都能用结构化的方式写出来，开发者就更容易调整它们。你不必每次都从运行时命令倒推界面结构，也不必在重构时担心某个节点被创建在远处、销毁在另一个函数里。

&emsp;&emsp;但声明式也会带来新的问题：当结构变化时，旧节点如何复用？子节点没有 key 时应该怎样提示？宿主节点重建时，原来的孩子是移动过去，还是销毁重建？动作和状态更新如何避免互相踩踏？这些问题如果处理不好，TSX 就会从“透明描图纸”变成“半透明迷雾”。

## 一个 useRef 小故事：兼容不是纵容

&emsp;&emsp;今年 DoraX 里有一个很适合写成小故事的改动：`useRef()` 在函数组件外被调用时应该怎么办。严格来说，Hook 应该只在函数组件里使用。这个规则清楚，也容易解释。如果照着最纯粹的设计，函数组件外调用 `useRef()` 就应该直接报错。

&emsp;&emsp;但真实项目不总是生活在纯粹规则里。DoraX 已经有一些历史写法，开发者可能把 `useRef()` 当作创建引用对象的便捷入口。如果我们突然把它全部变成错误，就会让旧代码在升级时被不必要地打断。于是问题变成：要不要为了规则的纯度牺牲兼容？还是为了兼容让规则变得含糊？

&emsp;&emsp;最后的选择很 Dora：`useRef()` 在组件外给出警告，并退回到普通 `reference()`；而 `useSignal()` 继续保持严格，只能在函数组件里使用。这个差异很重要。引用对象可以被看作比较轻的兼容层，状态信号却和组件生命周期紧密相关，不能随便脱离上下文。换句话说，兼容不是纵容，而是判断哪条规则可以留缓冲，哪条边界必须清楚。

&emsp;&emsp;这类决定看起来不像大功能，却会决定一个框架的性格。DoraX 不是只对新代码友好，也要尊重旧项目；但它也不能为了旧项目把未来的模型弄模糊。一个好的开源工具，常常就在这些小岔路上形成自己的分寸感。

## 子节点、key 和运行时的记忆

&emsp;&emsp;TSX 让场景结构更容易读，但运行时还需要“记住”哪些东西可以延续。比如一组动态子节点如果没有 key，表面上只是少写了一个属性，实际可能让框架很难判断哪个节点对应旧的哪个节点。网页 UI 里这是常识，游戏场景里同样成立，只是后果有时更隐蔽。节点可能带着动作、状态、物理关系或事件绑定，一次错误复用就会变成很难解释的行为。

&emsp;&emsp;因此 DoraX 对动态 children 的警告不是吹毛求疵。它像在工坊里提醒你给零件贴标签。今天桌上只有三个零件，你觉得不贴也能认出来；明天顺序变了、数量变了、其中一个还带着未完成的动作，问题就来了。key 的意义不是满足框架洁癖，而是让运行时有足够信息保存正确的记忆。

&emsp;&emsp;宿主节点重建也是类似故事。一个节点因为类型或关键属性变化不得不重建时，它下面的孩子怎么办？如果无脑销毁，用户状态可能丢；如果无脑保留，层级关系可能错。DoraX 在这里要做的不是追求某个抽象原则，而是在 Dora 场景树的真实行为里找到最不惊扰开发者的路径。声明式框架最难的地方正在这里：写起来像描述，运行时却必须是细致的搬运工。

## TSX 让 AI 更容易“看懂”场景

&emsp;&emsp;今年 AI 参与 Dora 开发以后，DoraX 的意义又多了一层。TSX 不只是让人类读起来舒服，也让 AI 更容易看懂项目意图。命令式场景代码像一条长长的施工记录，AI 需要在许多调用之间推断结构；TSX 则更像一张平面图，层级、属性、条件和组件边界都更直观。

&emsp;&emsp;这不是说 AI 只能写 TSX，而是说结构化表达会放大 AI 的可靠性。类型声明能检查属性，组件边界能限制修改范围，声明式层级能减少误解。对于一个希望让 Codex 或 Dora Agent 参与开发的引擎来说，这些都不是风格问题，而是工程反馈问题。AI 最怕的是含混上下文；TSX 恰好能把一部分上下文显式写出来。

&emsp;&emsp;从这个角度看，DoraX 不只是一个前端式语法糖，而是 Dora 面向 AI 协作的一条重要路径。它让场景树变得更可读，也让工具更容易围绕类型、结构和状态提供帮助。

## 小框架的难处是每个警告都要有意义

&emsp;&emsp;DoraX 的测试里还有一个很现实的细节：当 `useRef()` 在组件外进入兼容模式以后，测试套件必须允许这条预期警告。否则框架明明按设计提醒用户，自动化检查却会把它当成失败。这个动作很小，却说明一个框架的警告系统不能随便喊话。每条警告都应该有来处，也应该能被测试理解。

&emsp;&emsp;如果警告太随意，开发者会学会忽略它；如果测试不区分预期警告和异常警告，维护者又会为了通过测试把有用提示关掉。DoraX 在这里选择把警告纳入验证规则，本质上是在维护工具和用户之间的信用。你看到一条警告，应该相信它有意义；测试看到一条警告，也应该知道它是不是设计的一部分。
