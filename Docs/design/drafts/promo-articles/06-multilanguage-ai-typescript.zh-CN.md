# 给创作者留很多扇门：Dora 的多语言生态与 AI 时代的 TypeScript

> 「Dora SSR 2026 开发纪事」系列 · 第 6 篇 / 共 10 篇

## 一座有很多门的工坊

&emsp;&emsp;Dora SSR 支持 Lua、TypeScript、TSX、Teal、YueScript、Wa、Rust、C# 等多种语言。每次介绍这件事，都会遇到一个自然的问题：一个开源游戏引擎为什么要支持这么多语言？维护起来不累吗？

&emsp;&emsp;当然累。多一种语言，就多一套绑定、多一批示例、多一组文档、多一些构建路径，也多一些用户会遇到的边界问题。语言生态不是彩色贴纸，贴上去就结束；它更像给同一座工坊开很多扇门。门越多，来的人越多，风也越容易灌进来，门轴也更容易坏。

&emsp;&emsp;但 Dora 仍然想保留这些门。因为游戏开发是一种创作活动，创作者的表达习惯并不统一。有人喜欢 Lua 的轻快，有人喜欢 TypeScript 的类型反馈，有人喜欢 YueScript 的语法表达，有人想用 Rust 或 Wa 探索更底层的扩展，有人从 C# 生态过来更自然。如果工具只承认一种入口，它会变得整齐，但也会把很多人挡在外面。

## 多语言不是平均用力

&emsp;&emsp;不过，多语言并不意味着每种语言都要承担同样角色。Lua 适合作为轻便脚本层，TypeScript/TSX 适合更强的类型组织和结构化场景表达，YueScript 提供更高表达力的脚本风格，Wa、Rust、C# 则分别连接不同生态和性能边界。Dora 要做的不是把每种语言都变成全能工具，而是让它们围绕同一个引擎核心形成不同表面。

&emsp;&emsp;这个观点很重要。很多多语言项目失败，不是因为语言多，而是因为每种语言都像一套独立系统，彼此没有共同地基。Dora 的共同地基是引擎 API、资源模型、运行时行为和 Web IDE 工作流。语言只是不同入口，进入以后看到的应该仍然是同一座工坊。

&emsp;&emsp;这也是绑定和声明生成工作如此重要的原因。它们像门框，决定不同语言进入时是否顺手。门框歪一点，用户每天都会撞到。类型名、工厂函数、命名空间、旧写法兼容、新写法体验，这些看似细碎的问题，其实就是语言生态的手感。

## 一个 Node 里的小型语言史

&emsp;&emsp;今年关于 `Node` 的 TypeScript 声明讨论，是一个很能说明问题的小故事。用户自然会希望写出这样的代码：导入 `Node`，把它既当类型用，又当工厂函数调用。对人来说，这很合理：Node 就是节点，`Node()` 创建节点，`const node: Node` 表示一个节点。语言直觉是统一的。

&emsp;&emsp;但 TypeScript 的类型空间和值空间并不完全按人的直觉折叠。Dora 旧的声明形状为了兼容 Teal/TSTL 的生成方式，保留了 `NodeType`、`Node.Type` 和可调用工厂导出。它能工作，但要求用户在“Node 作为值”和“Node 作为类型”之间绕一下。想让 `const node: Node = Node()` 成立，就不能简单把同名 class 和 const 硬叠在一起，因为 class 已经同时占住类型和值空间。

&emsp;&emsp;于是这件事变成了一次小型语言考古。先看当前声明从哪里来，再用最小 TypeScript 片段确认错误不是想象出来的，再试不同声明形状，最后发现 `interface Node` 加同名 callable const，再保留 `namespace Node { type Type = Node }`，可以同时支持新写法和旧的 `Node.Type`。这不是“修一个类型报错”那么简单，而是在三种东西之间找平衡：人的直觉、TypeScript 的规则、旧项目的兼容。

&emsp;&emsp;这个故事很好地说明了多语言生态的真实成本。你不是只把 C++ API 包一层就结束，还要尊重目标语言自己的语法和习惯。用户写代码时希望自然，编译器却只承认规则。绑定维护者的工作，就是在自然和规则之间架桥。

## AI 会抹平一部分语言门槛

&emsp;&emsp;今年最大的背景是 AI。Codex 参与 Dora 开发以后，我们越来越明显地感觉到，多语言支持的意义正在变化。过去多语言主要解决“人喜欢用什么语言”的问题；未来它还会解决“AI 能把人的意图翻译到哪种语言表面上”的问题。

&emsp;&emsp;一个用户可能并不熟悉 YueScript，但他可以让 Agent 解释一段旧代码；他可能不熟悉 Dora 的 TypeScript 声明，但可以让 Agent 根据类型提示生成结构；他可能只描述一个玩法原型，由 Agent 选择更适合的脚本入口。AI 不会让语言差异消失，但会把很多入门门槛削低。过去必须先读语法、查 API、理解构建路径，现在可以由 AI 先搭出第一座桥。

&emsp;&emsp;不过，AI 抹平门槛并不意味着底层语言生态可以随便。恰恰相反，AI 越参与，语言表面越需要清晰。类型声明越准确，AI 越容易生成可检查代码；错误信息越具体，AI 越容易修；项目入口越稳定，AI 越不容易猜错。AI 不是替代工具质量，而是放大工具质量。

## 为什么当前最优解会落在 AI + TypeScript

&emsp;&emsp;在 Dora 的多语言生态里，TypeScript 今年变得尤其关键。原因不是语言信仰，而是它提供了 AI 协作特别需要的反馈。类型声明告诉 AI 哪些 API 存在，语义诊断告诉它哪里不成立，TSX 结构让场景树更容易被读懂，Monaco 编辑器能在实时编辑阶段给出轻量反馈，TSTL 转译路径又能在构建阶段提供更严格检查。

&emsp;&emsp;这使得 TypeScript 像一条中轴路。其它语言仍然重要，Lua 的直接、YueScript 的表达、Wa/Rust/C# 的边界探索都不会消失；但当我们谈“AI 参与游戏开发的默认体验”时，TypeScript 暂时拥有最好的工具反馈密度。AI 可以写，人可以读，编辑器可以查，构建可以验，类型声明可以沉淀引擎知识。

&emsp;&emsp;这也是 Dora 今年聚焦 AI + TypeScript 的原因。它不是要关闭其它门，而是先把最适合 AI 协作的主路铺平。主路越稳，其它门也越容易通过 Agent 被连接起来。
