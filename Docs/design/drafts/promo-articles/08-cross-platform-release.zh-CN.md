# 从 GitHub 到掌机：Dora 的跨平台发布像一次港口调度

> 「Dora SSR 2026 开发纪事」系列 · 第 8 篇 / 共 10 篇

## 发布不是按下一个按钮

&emsp;&emsp;对外看，一个开源项目发布新版本似乎很简单：打一个 tag，等 CI 跑完，把压缩包放到 release 页面，写几行更新说明。真正做过跨平台发布以后就会知道，这更像一次港口调度。不同平台的包是不同船只，GitHub、Gitee、GitCode、PPA、用户下载目录是不同码头，tag、版本号、发布说明、资产大小、哈希和构建状态是航运单据。任何一个环节对不上，用户拿到的就可能不是你以为的那个版本。

&emsp;&emsp;Dora SSR 今年的发布工作里，很多精力花在这种“不浪漫但重要”的确认上。版本脚本有没有跑？应用元数据有没有更新？tag 指向的是不是新的 commit？GitHub Actions 的三个平台包是否全部完成？Gitee 上同名附件是不是旧的？GitCode 的上传接口有没有真正返回成功？下载后的文件大小和源资产是否匹配？这些问题没有一个像新功能那样令人兴奋，但它们决定了开源项目对用户是否可信。

&emsp;&emsp;发布工作最怕一句话：“应该已经是最新的。”Dora 今年越来越明确地不相信这句话。最新必须被验证，不能被假设。

## 一个“最新包”的小故事

&emsp;&emsp;有一次发布同步的会话很典型。表面任务只是把最新包同步到各个平台，但“最新”这个词本身就是陷阱。tag 名字没有变，不代表资产没变；本地下载目录里有文件，不代表它来自最新的 GitHub Release；GitCode 或 Gitee 上出现同名附件，不代表它已经被覆盖成功。

&emsp;&emsp;于是流程被拆成一连串看似笨拙的动作：先读 live GitHub release，而不是看本地草稿；再下载当前资产，而不是复用缓存；检查文件大小和哈希；如果下游平台有同名附件，先删除再上传；上传以后再读回状态；最后把确认过的包移到用户指定的位置。这个过程慢、笨拙，每一步都在反复确认，却正是不容易出错的原因。

&emsp;&emsp;这里面有一个很适合开源项目的观点：发布不是把文件推出去，而是让用户能拿到你承诺的文件。前者是动作，后者是责任。Dora 做跨平台发布时，真正要维护的是这种责任链。

## 多平台不是一句“支持”

&emsp;&emsp;Dora SSR 面向多个平台：桌面、Android、Linux、ARM64 设备、开源掌机，以及通过包管理器或发布页进入用户机器的各种路径。每个平台都有自己的交付习惯。macOS 包要有自己的应用元数据，Windows 包有资源版本，Android 有版本号，Linux 用户可能更希望通过 PPA 安装，掌机用户则可能从 Ports 菜单启动。

&emsp;&emsp;这让“跨平台”不再是编译成功四个字。真正的跨平台发布，必须把每个平台用户的到达路径考虑进去。一个 Linux ARM64 包如果只能在 CI 里生成，却不能被目标设备方便安装，它离用户还很远；一个掌机上能运行的二进制，如果没有接到前端菜单，用户仍然不知道怎样启动；一个 GitHub Release 如果同步不到国内可访问的平台，部分用户就会被网络环境挡在门外。

&emsp;&emsp;所以 Dora 的发布流程越来越像分发网络，而不是单点上传。GitHub 是源头，其他平台是延伸，PPA 是 Linux 用户的系统入口，设备上的启动脚本是最后一米。每一米都要有人确认。

## 发布说明也是产品的一部分

&emsp;&emsp;发布说明常常被当成最后补上的文字，但它其实是用户理解版本的第一扇窗。Dora 今年也处理过直接修改 live GitHub release body 的任务。这里的教训很简单：如果公共页面上的文字不对，改本地文件没有意义；真正的源头是用户看到的 release 页面。改完以后还要读回来确认，否则你只是相信自己已经改了。

&emsp;&emsp;这听起来像小事，但它和代码发布是同一种原则：用户接触到的东西才是事实。仓库里的草稿、命令行里的输出、本地缓存里的 zip，都不如 live 页面和实际下载资产重要。开源项目尤其需要这种意识，因为用户常常不会从维护者脑子里的流程理解版本，只会从页面和包本身理解。

&emsp;&emsp;好的发布说明也不只是列功能。它应该告诉用户这个版本为什么值得升级，哪些问题被修复，哪些平台路径更稳，哪些行为可能影响旧项目。

## 发布流程里的克制

&emsp;&emsp;发布工作有时会让人焦躁。CI 排队、远端接口失败、GitCode 连接不上、Launchpad 状态还在 pending、某个附件列表没有预期字段。这时最容易犯的错是不断重复同一个命令，仿佛多刷新几次就能让事实改变。Dora 的会话记录里也沉淀出一个经验：轮询要有边界，确认关键状态后就停，不要让验证本身变成噪音。

&emsp;&emsp;还有一个经验是承认环境边界。某些远端在当前网络或凭据环境下就是不好用，如果用户已经手动处理 GitCode，同步流程就不应该继续被它拖住。工程协作不是证明 AI 能控制所有平台，而是知道什么时候把任务拆开，什么时候接受外部状态已经由人接手。

&emsp;&emsp;这种克制和发布可靠性并不矛盾。恰恰相反，可靠的发布流程需要知道哪些事情必须验证，哪些事情不该无限消耗时间。港口调度不是每条船都亲自开到终点，而是确保关键交接有记录、有回读、有责任边界。

## 忘记跑版本脚本以后

&emsp;&emsp;发布流程里还有一种更常见的事故：不是构建失败，而是某个版本元数据没有更新。Dora 之前就遇到过需要重新发布的情况，原因不是代码不能跑，而是发布前忘了跑版本更新脚本。这个错误很像给货箱换了新货，却忘了更新箱外标签。用户下载以后看到的版本号、应用元数据或系统显示，可能和维护者以为的不一致。

&emsp;&emsp;这次之后，发布会话里形成了一条很实用的规则：如果用户说上一次发布漏了版本脚本，下一次就不要再假设元数据已经正确。先跑版本更新，确认修改，提交，再移动 tag，等 GitHub Actions 重新产出资产，然后再同步下游平台。流程看起来变长了，但它把“人记得”变成了“流程记得”。

&emsp;&emsp;开源项目的发布可靠性，很多时候就是把这种记忆写进步骤。维护者可以累，可以忘，AI 可以把检查项重新摆出来。港口每天都忙，靠脑子记住每个箱号并不现实，靠清单才现实。
