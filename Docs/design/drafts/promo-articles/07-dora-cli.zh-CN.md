# Dora CLI：一个开源游戏引擎的入口应该像路标

> 「Dora SSR 2026 开发纪事」系列 · 第 7 篇 / 共 10 篇

## 命令行不是退回旧时代

&emsp;&emsp;一提到游戏引擎，很多人首先想到的是图形化编辑器。菜单、面板、资源浏览器、场景视图、属性检查器，这些当然是现代开发工具的重要部分。但对 Dora SSR 来说，CLI 同样重要。它不是对图形界面的退让，也不是给“高级用户”的隐藏通道，而是整个工作流的路标。

&emsp;&emsp;路标的价值在于清楚。你站在项目门口，不应该先猜服务有没有启动、Web IDE 在哪个端口、项目目录是不是正确、构建该跑哪条命令。CLI 要把这些问题用最直接的方式摆出来。它不需要华丽，只需要可信。一个好的命令行工具像一块干净的指示牌：往这边走能打开编辑器，往那边走能检查环境，遇到问题先看这里。

&emsp;&emsp;Dora SSR 今年围绕 CLI 做的改进，很多都不是“新增一个炫酷命令”，而是让入口变得更稳、更能解释自己。对一个开源游戏引擎来说，这比想象中更重要。因为用户第一次遇到问题时，通常不会去读源码，也不会理解内部服务结构。他只会想：我现在该输入什么？

## status 和 doctor 的分工

&emsp;&emsp;Dora CLI 里很有代表性的设计，是 status 和 doctor 的分工。status 应该短，像门口的信号灯，告诉你现在是不是通；doctor 应该细，像修理台上的检查单，告诉你哪里可能坏了，能不能自动修复。把这两者混在一起，用户要么被冗长信息淹没，要么在真正出问题时得不到足够线索。

&emsp;&emsp;这两条命令打印出来的差别，就是这个观点最直接的体现。`dora cli status` 只回答一件事——服务通不通：

```
Native engine: yes
Dora SSR: <版本号> on <平台>
Web IDE URL: http://127.0.0.1:8866
Web IDE connected: no (0)
Running: no
```

而 `dora cli doctor` 才像打开工具箱，按 CLI、Project、Service、Tooling 四块展开，连 `Wa template` 这种平时用不上、排查时才需要的细节也只放在这里，不污染 status：

```
CLI:
  Executable: /usr/bin/dora-ssr
  CWD: /home/user/mygame
Project:
  Path exists: yes
  Is directory: yes
  Init files: init.lua
  Script dir: yes
  tsconfig.json: yes
  API dir: yes
Service:
Native engine: yes
Dora SSR: <版本号> on <平台>
Web IDE URL: http://127.0.0.1:8866
Tooling:
  Wa template: no
```

&emsp;&emsp;这其实是工具设计里一个很小但很重要的观点：同一个事实，在不同场景下需要不同密度。你每天启动项目时，只想知道“能不能用”；当它不能用时，才需要看到端口、服务、路径、依赖、Web IDE、修复入口这些细节。CLI 如果总是把所有信息倒出来，就像每次开门都递给你一份建筑图纸，久了只会让人疲惫。

&emsp;&emsp;Dora 的 doctor 还承担了另一个任务：把“知道问题”和“修复问题”接近一些。它不仅检查 Dora 服务状态，也能通过修复路径帮助启动本地引擎、打开 Web IDE、确认项目资源目录。这让命令行不只是诊断仪，而更像一个带工具盒的向导。

## CLI 是 Web IDE 的地基

&emsp;&emsp;很多用户看到 Web IDE，会觉得它就是 Dora 的入口。但从工程上看，Web IDE 能不能稳定工作，很大程度依赖 CLI 和本地服务提供的地基。浏览器里打开的界面需要知道项目在哪里，服务是否健康，构建接口是否可用，文件系统能不能访问，运行时是否能响应。

&emsp;&emsp;这就是为什么 Dora 的 CLI 和 WebServer 之间有清晰的接口。CLI 不是孤立地打印文字，而是通过本地服务检查状态、请求修复、打开编辑环境。用户输入的是一个命令，背后串起来的是项目目录、Dora 运行时、Web IDE 页面和构建工具链。这个链条越稳定，用户越不需要理解链条本身。

&emsp;&emsp;开源工具常常在这里吃亏：功能已经存在，但入口太散。文档说一条命令，仓库脚本又有另一条，本地服务报错再去看第三个位置。Dora CLI 的目标就是把这些入口尽量收束，让开发者从一开始就站在一条可走的路上。

## 多语言构建需要一个统一门口

&emsp;&emsp;Dora 支持多种脚本和资源形式，这也让 CLI 的角色更重。TypeScript、YueScript、Teal、XML、Wa、Yarn 等不同输入，背后有不同的检查和生成逻辑。如果每一种都要求开发者记住独立流程，Dora 的多语言优势就会被工具链复杂度抵消。

&emsp;&emsp;所以 CLI 需要像车站大厅，把不同线路接到同一个入口。开发者不一定要知道每条线路背后的转辙机如何工作，但他需要知道从哪里进站、出错时哪里显示原因、什么时候该重新构建。这个统一入口对人有帮助，对 AI 也有帮助。因为 Agent 要参与项目时，最怕的就是每个任务都要重新猜测验证命令。

&emsp;&emsp;一个稳定 CLI 对 AI 来说像一条可复用的脚印。今天修 Agent 代码，知道应该跑哪个构建；明天检查 Yarn 文件，知道 CLI 会把语法问题纳入验证；后天处理项目状态，知道先看 status 还是 doctor。工具越能自我说明，AI 越不容易在项目里迷路。

## 好的入口会降低恐惧感

&emsp;&emsp;新用户面对一个开源引擎时，最先感受到的往往不是功能强弱，而是恐惧感。这个项目怎么启动？失败了是不是我环境错了？文档是不是过期了？报错是正常现象还是我哪里做错了？如果入口混乱，这种恐惧会迅速放大，用户可能还没看到引擎真正能力，就已经离开了。

&emsp;&emsp;CLI 的意义就在这里。它把很多不确定性变成可询问的状态。你可以问 Dora：你现在好吗？你能不能修？你能不能打开项目？这听起来像拟人化，但对工具体验很真实。一个愿意回答自己状态的引擎，比一个沉默失败的引擎更容易让人信任。

&emsp;&emsp;Dora SSR 不是要把所有复杂性藏起来。开源项目也不应该假装没有复杂性。更好的方式是把复杂性放到合适的位置：日常入口简单，诊断入口详细，修复入口明确，文档入口可追踪。CLI 就是这些入口之间的路标系统。

## 一次关于“太多信息”的小争论

&emsp;&emsp;CLI 的 doctor/status 会话里，有一段很像真实产品设计讨论。最开始我们希望把状态信息尽量补全，于是服务状态、项目路径、工具链、Wa template 是否准备好都可能被放进输出里。信息越多，好像越负责任。但用户很快指出：`status` 不用展示 `Wa template: yes`。

&emsp;&emsp;这句话表面上是在删一行输出，实际上是在给命令定义性格。`status` 应该像门口的信号灯，短、稳定、适合脚本读取；`doctor` 才像维修师傅打开工具箱，可以慢慢展开 CLI、Project、Service、Tooling 几个部分。把 Wa template 这种细节塞进 status，会让信号灯变成公告栏。第一次看很详细，第二十次就只剩噪音。

&emsp;&emsp;后来 `doctor --fix` 也经历过类似收窄。修复命令当然应该能帮用户重新连接本地 Web IDE，但它不能在非本地场景里过度猜测，也不应该由 CLI 自己乱开浏览器。最终更合理的做法，是让本地服务确认上下文，再通过引擎侧打开 Web IDE。这个过程说明：好入口不只是功能多，而是每个入口知道自己该管到哪里为止。

## CLI 也要避免吵醒整座引擎

&emsp;&emsp;另一个小故事来自 `--asset`。用户希望 `dora cli` 可以和 `--asset` 一起使用，让命令行检查指定资源目录。最直接的实现很容易把引擎内容系统初始化起来，但这样做会带来额外副作用，退出时还会出现本不该出现的销毁日志。对一个只想问状态的 CLI 来说，这就像为了看门牌号把整栋房子灯都打开。

&emsp;&emsp;最后的方向是保持 CLI 轻量：解析 `--asset`，把它作为最小的上下文传给 Lua 侧，而不是唤醒完整引擎单例。这个细节和 status/doctor 的分工是同一种思想。命令行入口越基础，越要克制。它应该能看清路，但不该每走一步都惊动整个系统。
